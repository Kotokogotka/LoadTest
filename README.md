# FormLoadTests

Нагрузочные тесты на `Locust` для сценария входа в систему и открытия анкеты через HTTP API.

Сейчас в проекте реализован один основной поток:

- виртуальный пользователь берет свободный тестовый аккаунт из общего пула;
- выполняет логин через `/auth-be/auth/jwt/login`;
- открывает случайную анкету из списка `FORM_IDS`;
- при остановке выполняет logout и возвращает аккаунт обратно в пул.

## Стек

- Python 3.10+ (`venv` в проекте создан на Python 3.14)
- `locust`
- `pydantic`
- `keyring`

## Структура проекта

- `locustfiles/main.py` — точка входа для `Locust`, инициализация пула аккаунтов и основной `HttpUser`.
- `accounts/loader.py` — загрузка списка аккаунтов из `secrets/test_users.json` и паролей из `keyring`.
- `accounts/pool.py` — потокобезопасный пул тестовых аккаунтов.
- `clients/auth_client.py` — login/logout запросы.
- `scenarios/user_flows.py` — текущий пользовательский сценарий: открытие случайной анкеты.
- `scripts/seed_keyring.py` — массовая запись общего пароля в `keyring` для алиасов `u001`-`u010`.
- `scripts/read_users.py` — вспомогательная загрузка пользователей вместе с паролями из `keyring`.

## Подготовка

### 1. Установить зависимости

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

### 2. Подготовить список тестовых пользователей

Проект ожидает файл `secrets/test_users.json` со структурой:

```json
[
  {
    "alias": "u001",
    "username": "user1@example.com",
    "rid": "267"
  },
  {
    "alias": "u002",
    "username": "user2@example.com",
    "rid": "195"
  }
]
```

Поля:

- `alias` — ключ для поиска пароля в `keyring`;
- `username` — логин пользователя;
- `rid` — дополнительный идентификатор аккаунта, который хранится в runtime-модели.

Каталог `secrets/` добавлен в `.gitignore`, поэтому хранить в нем локальные тестовые данные допустимо.

### 3. Записать пароли в `keyring`

Пароли не читаются из JSON-файла. Для каждого `alias` проект берет пароль из системного `keyring` с сервисом `form-load-tests`.

Если у всех тестовых пользователей один и тот же пароль, можно использовать готовый скрипт:

```bash
python3 scripts/seed_keyring.py
```

Скрипт запросит пароль и сохранит его для алиасов `u001`-`u010`.

Важно:

- алиасы в `keyring` должны совпадать с `alias` в `secrets/test_users.json`;
- если пароль для алиаса не найден, запуск завершится ошибкой;
- если аккаунтов меньше, чем одновременно стартующих пользователей `Locust`, часть пользователей упадет на получении аккаунта из пула.

## Запуск

Хост приложения не зашит в код и передается через параметр `--host`.

### Веб-интерфейс `Locust`

```bash
locust -f locustfiles/main.py --host https://example.test
```

После запуска UI обычно доступен по адресу `http://localhost:8089`.

### Headless-режим

```bash
locust -f locustfiles/main.py \
  --host https://example.test \
  --headless \
  --users 8 \
  --spawn-rate 2 \
  --run-time 5m
```

## Как работает сценарий

1. На этапе `events.init` загружается список аккаунтов и создается общий `AccountPool`.
2. В `ApiUser.on_start()` виртуальный пользователь получает свободный аккаунт и логинится один раз.
3. В задаче `open_form()` вызывается `visit_random_form()`, которая открывает одну случайную анкету из заранее заданного списка ID.
4. В `ApiUser.on_stop()` выполняется logout, после чего аккаунт возвращается в пул.

Список анкет сейчас захардкожен в `scenarios/user_flows.py`:

```python
FORM_IDS = [116, 223, 119, 165, 183, 224, 255, 232, 141]
```

## Текущее ограничение

На данный момент в проекте реализован только сценарий открытия анкеты после логина. Файлы `scenarios/auth_flows.py`, `scenarios/admin_flows.py` и `clients/users_client.py` пока выступают как заготовки для следующих сценариев.
